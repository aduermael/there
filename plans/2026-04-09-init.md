# Game Stack POC — Implementation Plan

**Date**: 2026-04-09
**Goal**: Open a URL, you're on terrain. Share the URL, your friend sees you. Desktop + mobile Safari. That's it.

---

## Architecture Overview

```
┌─────────────────────────────────────────────────┐
│                  game-core/                      │
│  Shared Rust crate: types, movement, validation  │
│  (no GPU, no networking, no platform deps)       │
└──────────┬──────────────────┬────────────────────┘
           │                  │
    ┌──────▼──────┐    ┌──────▼──────┐
    │ game-client │    │ game-server │
    │ Rust→WASM   │    │ Rust binary │
    │ wgpu WebGPU │    │ axum + tokio│
    │ Browser     │    │ Headless    │
    └──────┬──────┘    └──────┬──────┘
           │                  │
           │   WebSocket      │
           └──────────────────┘
```

**Single server binary** serves static files (embedded via `rust-embed`) AND handles WebSocket on the same port. Zero config deployment.

## UI Philosophy

All UI is **plain HTML/CSS web components layered on top of the WebGPU canvas**. The canvas is a fullscreen surface that only handles 3D rendering. Everything else — HUD, menus, joystick overlays, room codes, debug info — lives in the DOM as simple web components positioned over the canvas with CSS.

This gives us:
- Standard web primitives for all UI (flexbox, transitions, media queries, accessibility)
- Easy styling and iteration without touching Rust/WASM
- Touch events naturally handled by the DOM layer, passed to WASM only when needed
- Mobile-responsive UI for free via CSS
- Clean separation: WASM renders the world, the browser renders the UI

```
┌──────────────────────────────────┐
│  DOM Layer (web components)      │  ← HUD, joystick, menus, room code
│  ┌────────────────────────────┐  │
│  │  <canvas> WebGPU surface   │  │  ← 3D world only
│  └────────────────────────────┘  │
└──────────────────────────────────┘
```

The canvas sits at `z-index: 0`, UI components float above. Pointer events pass through transparent regions to the canvas via `pointer-events: none` on non-interactive UI areas.

## Stack Decisions

| Layer | Choice | Why |
|-------|--------|-----|
| **Rendering** | `wgpu` 24.x → WebGPU | Only serious Rust WebGPU option. Maps to Metal on iOS. |
| **ECS** | `hecs` | Tiny footprint. Bevy is 10+ MB WASM — disqualified. |
| **Math** | `glam` | Same lib wgpu uses internally. ~0 cost. |
| **Serialization** | `bincode` over `serde` | Position update = 15 bytes vs ~120 for JSON. |
| **Server** | `axum` on `tokio` | First-class WebSocket, serves static files too. |
| **Transport** | **WebSocket (binary)** | Only option that works on iOS Safari. WebTransport has no Safari support. WebRTC is overkill. |
| **WASM build** | `wasm-bindgen` + `wasm-opt` | wasm-pack wrapper is fine too. Target: < 2 MB .wasm. |
| **Touch input** | `nipplejs` (~4 KB) | Battle-tested virtual joystick. Left stick = move, right area = camera drag. |
| **UI** | HTML/CSS web components over canvas | No in-engine UI. All UI is DOM-based for maximum flexibility. |

## Terrain Rendering

### Heightmap
- **512x512 heightmap**, world = 256m x 256m (0.5m per texel)
- **GPU vertex displacement**: flat grid mesh generated on CPU (XZ positions + UV), heightmap sampled in vertex shader via `textureSampleLevel`
- Future terrain modification just means updating the heightmap texture — no mesh rebuild

### Chunks & LOD
- **16x16 chunks** of 32x32 vertices each
- **2 LOD levels**: full-res + half-res index buffers, distance-based swap
- Skirt geometry on chunk edges to hide LOD cracks (drop edge verts down)
- Frustum culling per chunk
- ~65K vertices visible at any time, halved with LOD

### Texturing
- **Height-based color lerp** in fragment shader: sand → grass → rock based on Y
- 1 tiling detail noise texture multiplied in to break flatness
- Total: 2 texture binds per terrain draw (heightmap + detail noise)
- No splat maps for POC

### Lighting
- 1 directional light (sun) + 0.3 constant ambient
- Terrain normals computed in vertex shader from heightmap finite differences (sample 4 neighbors)
- Standard `N dot L` diffuse
- Distance fog: `mix(color, fogColor, saturate(dist/farPlane))` — adds depth, hides LOD pops
- No shadows for POC

### Draw Call Budget
- Target: **< 30 draw calls total**
- Terrain: 1 shared pipeline, 1 shared vertex buffer (grid template), per-chunk offset via uniform. ~20 chunk draws visible
- Players: 1 instanced draw call for all capsules
- Sky/fog: 1 fullscreen quad
- All chunks share same pipeline + bind group

## Third-Person Camera

- Spherical coordinates: yaw, pitch, distance relative to player
- Smooth follow: `lerp(current, target, dt * speed)` each frame
- Collision: single raycast from player to desired camera position against terrain heightmap (sample height at ray XZ, compare Y). Clamp on hit
- No physics engine needed

---

## Player Representation

- **Capsule** (cylinder + 2 hemispheres): ~200 triangles, procedurally generated at startup
- Flat-shaded, single color per player (assigned on join)
- Rendered instanced — 1 draw call for all players

## Networking

### Transport
- **WebSocket (binary frames)** over TLS (`wss://`)
- Works universally: Chrome, Firefox, Safari desktop, Safari iOS
- Latency overhead vs UDP: ~1-5ms from TCP head-of-line blocking — negligible for 2-8 players

### Server
- **Axum** on Tokio async runtime
- Single binary handles HTTP (static files) + WebSocket upgrades on same port
- Rooms stored in `HashMap<String, Room>` behind `Arc<RwLock<>>`
- No database, no external dependencies

### Server-Authoritative Model
- **20 Hz tick** (50ms intervals)
- Server receives `PlayerInput { forward, strafe, yaw }`
- Validates: speed clamp, boundary check, terrain collision via `game-core`
- Broadcasts `WorldSnapshot` to all players in room

### Wire Protocol
Hand-rolled binary via shared `game-core` enums:

```
ClientMsg::Input { forward: f32, strafe: f32, yaw: f32 }   // 13 bytes
ServerMsg::Snapshot { players: Vec<(u16, f32, f32, f32, f16)> }  // 15 bytes/player
```

Serialized with `bincode`. On the JS side, `DataView` over `ArrayBuffer` for the boot/glue layer; most protocol handling happens in WASM.

### State Sync
- **Full state broadcast every tick**. With 2-8 players at ~15 bytes each = ≤120 bytes/tick
- No delta compression, no snapshot interpolation — overkill at this scale

### Latency Hiding
- **Client prediction**: apply local movement immediately on input, snap-correct when server state arrives (lerp if delta is small)
- **Other-player interpolation**: buffer 2 most recent snapshots, render remote players interpolated between `t-2` and `t-1` (one tick behind = 50ms visual delay, imperceptible for walking)

## Room Joining

1. User visits `https://host/` → server generates 4-char alphanumeric code (36^4 = 1.7M combinations), redirects to `/ABCD`
2. `index.html` loads, JS extracts room code from URL path
3. WebSocket connects to `wss://host/ws?room=ABCD`
4. First connect creates room + starts headless game instance; subsequent connects join
5. Share the URL → friend opens it → same room
6. Rooms auto-expire 60s after last player disconnects

No lobby. No login. No waiting screen.

## Mobile Safari (iOS 18+)

### WebGPU
- Enabled by default since Safari 18 / iOS 18 (shipped late 2024)
- Maps to Metal under the hood — native Apple GPU performance
- Feature-detect with `navigator.gpu`, show "game can't run" fallback otherwise
- Stick to core WebGPU spec — Safari lags Chromium on extensions

### WASM
- Full JIT enabled, streaming compilation via `WebAssembly.instantiateStreaming`
- Practical memory limit on iPhone: ~1-1.5 GB before OS kills the tab
- Keep .wasm binary < 2-3 MB. Use `wasm-opt -Oz` + Brotli/gzip

### Viewport & Fullscreen
```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
```
- iOS Safari has **no Fullscreen API** for canvas
- Ship PWA manifest (`"display": "standalone"`) — "Add to Home Screen" removes address bar entirely
- Without PWA, address bar auto-hides on scroll; start game after first touch to trigger

### Touch Controls (web components over canvas)
- **Left thumb**: nipplejs virtual joystick (movement) — rendered as a DOM overlay web component
- **Right thumb area**: drag for camera rotation — touch events on a transparent overlay, forwarded to WASM
- `touch-action: none` on the canvas to prevent browser gestures
- Dead zone of ~10% on joystick to prevent drift

### Orientation
- Lock to landscape via manifest `"orientation": "landscape"` for PWA
- In-browser: `screen.orientation.lock("landscape")` (may silently fail on iOS — show "rotate device" prompt via `matchMedia("(orientation: portrait)")`)

### Safe Areas & Responsive
- Canvas: `100dvh x 100dvw` (dynamic viewport units handle address bar)
- UI web components: use `env(safe-area-inset-*)` CSS to avoid notch / Dynamic Island / home indicator
- Test on both notched (iPhone 14) and Dynamic Island (iPhone 15+) devices

### Network
- iOS aggressively suspends background tabs — WebSocket disconnects within seconds of tab-switch
- Implement reconnect with exponential backoff
- Handle `visibilitychange` events to pause/resume gracefully

### Performance Budget
- **60 FPS** achievable on A14+ GPUs (iPhone 12 and newer)
- Budget: < 200K triangles/frame, < 200 draw calls
- Our terrain + capsules target: ~65K tris, ~30 draw calls — well within budget

## Worker Architecture — Deferred

WebGPU *can* run off main thread via `OffscreenCanvas` + worker in Chrome. **However, Safari's OffscreenCanvas + WebGPU-in-worker support is unreliable as of early 2026.**

**POC decision**: run rendering on main thread. Game logic is cleanly separated inside the WASM module anyway. Revisit worker offloading post-POC once Safari stabilizes.

## Shared Crate: `game-core`

Zero platform dependencies. No `wgpu`, no `tokio`, no `std::net`.

Contains:
- **Types**: `Vec3` (re-export glam), `PlayerId(u16)`, `RoomCode(String)`, `EntityId`
- **Movement**: `apply_movement(pos, input, dt, heightmap) -> Vec3` — runs identically on client and server
- **Terrain**: `sample_height(heightmap, x, z) -> f32`, boundary checks
- **Protocol**: `ClientMsg` / `ServerMsg` enums with bincode serialization
- **Constants**: `TICK_RATE_HZ = 20`, `MOVE_SPEED = 5.0`, `WORLD_SIZE = 256.0`, `HEIGHTMAP_RES = 512`

## Build & Dev Workflow

```bash
# Build client WASM
wasm-pack build game-client --target web --out-dir ../web/pkg

# Build server (release)
cargo build -p game-server --release

# Dev — two terminals:
# Terminal 1: server with auto-restart
cargo watch -w game-core -w game-server -x 'run -p game-server'

# Terminal 2: client WASM rebuild
wasm-pack build game-client --dev --target web --out-dir ../web/pkg
# Then refresh browser
```

No hot-reload for POC — manual browser refresh is fine.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Safari WebGPU quirks vs Chrome | Test on real iOS device from Phase 3. Use `textureSampleLevel` not `textureSample` in vertex shaders. Stick to `f32`. Avoid storage buffers. |
| WASM binary size bloat | No Bevy. Minimal deps. `wasm-opt -Oz`. Monitor size each phase. Target < 2 MB. |
| OffscreenCanvas broken on Safari | Skip worker architecture entirely for POC. Main thread rendering. |
| WebSocket disconnect on iOS background | `visibilitychange` + reconnect with exponential backoff + auto-rejoin room. |
| Heightmap precision on mobile | Use R16 PNG (16-bit) for heightmap. Test on real device early. |
| Touch input conflicts with browser gestures | `touch-action: none` on canvas + `user-scalable=no` viewport meta. |

## What's NOT in the POC

- Terrain modification (heightmap editing) — deferred, but architecture supports it (just update texture)
- Physics engine — movement is simple position + heightmap collision
- Authentication / accounts — room codes only
- Persistence — rooms are ephemeral
- Audio
- Shadows, water, vegetation, weather
- Worker-based rendering
- Compute shaders
